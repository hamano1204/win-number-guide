# WinNumberGuide 設計書

## 1. プロジェクト概要

**WinNumberGuide** は、Windows 11（およびWindows 10）環境において、Windowsキーを長押しした際にタスクバーにピン留め・起動されているアプリケーションのショートカット番号（1〜9、0）を画面中央にオーバーレイ表示するデスクトップアプリケーションです。
ユーザーは視覚的に「Winキー + 数字キー」の対応を確認でき、アプリケーションの切り替えや起動をスムーズに行うことができます。

## 2. システム要件・技術スタック

- **対象OS**: Windows 10 / Windows 11
- **フレームワーク**: .NET 10.0 (net10.0-windows10.0.22621.0)
- **UIテクノロジー**: WPF (Windows Presentation Foundation)
- **主要なWindows API・ライブラリ**:
  - `UIAutomationClient` (タスクバー情報の取得)
  - `user32.dll` (グローバルキーボードフック用ネイティブAPI)
  - `System.Windows.Forms.NotifyIcon` (システムトレイアイコン)
  - `Microsoft.Win32.Registry` (スタートアップ登録)
  - `Windows.Management.Deployment` (UWPアプリのアイコン取得用 PackageManager API)

## 3. アーキテクチャ・コンポーネント構成

本アプリケーションは、主に以下のコンポーネントから構成されています。

### 3.1. キーボード入力監視 (KeyboardHook.cs)
低レベルキーボードフック（`WH_KEYBOARD_LL`）を使用し、OS全体のキーボードイベントを監視します。
- Windowsキー（左/右）の押下状態を追跡します。
- Winキーが押された瞬間に `WinKeyDown` イベントを発火し、後述するプリフェッチを開始させます。
- Winキーが単独で押され、一定時間（600ms）保持された場合に `WinKeyLongPressed` イベントを発火します。
- 他のキーが押された場合や、Winキーが離された場合はタイマーをキャンセルし、UIを非表示にするためのイベントを発火します。

### 3.2. タスクバー情報取得 (TaskbarReader.cs)
Windowsの「UI Automation」を利用して、エクスプローラー（タスクバー）のUI要素ツリーを解析します。
- **高速化戦略 (TreeWalker)**: `FindAll(Descendants)` による全走査を避け、`FindFirst` でボタンを1つ見つけた後にその親コンテナを特定し、直接の子（`Children`）のみを列挙することで、プロセス間通信（IPC）のコストを大幅に削減しています。
- 各ボタンからアプリケーション名、AutomationId（AppId）を取得します。
- 取得した情報を元に、左から順に番号（1〜9, 0）を割り当て、表示用のデータモデル（`TaskbarApp`）を生成します。左右のキーボード配置を考慮し、5番目と6番目の間に余白（34px）を挿入するロジックも含みます。

### 3.3. アイコン抽出 (IconExtractor.cs)
タスクバー上の各アプリケーションからアイコンを抽出します。
- **効率的なキャッシュ**: 取得したアイコンを `Dictionary` でメモリ保持します。ただし、起動直後などで「アイコンが見つからなかった（null）」場合はあえてキャッシュせず、次回再スキャンを許容するフェイルセーフな設計としています。
- **非同期読み込み**: アイコン抽出（プロセス解析やUWPマニフェスト読み込み）はバックグラウンドスレッドで実行され、UIのメイン表示を妨げません。

### 3.4. ユーザーインターフェース (MainWindow.xaml / MainWindow.xaml.cs)
取得したタスクバーアプリケーションのリストを画面に表示するWPFウィンドウです。
- **コンパクトなUI**: 800px x 135px のサイズで設計され、デスクトップ上での視認性と邪魔にならないバランスを両立しています。
- **先行読み込みと即時表示 (Idea 1+3)**: Winキーが押された瞬間にバックグラウンドで情報の取得を開始（プリフェッチ）。長押しが確定した瞬間に、前回キャッシュがあればそれを即座に（0msで）表示し、準備が整った最新データで裏から差し替えることで、体感ラグをゼロにしています。
- **メモリ管理**: UIが非表示になる際（フェードアウト完了時）に `ItemsSource` をクリアし、不要なUIリソースを解放します。一方、アイコンのメモリキャッシュは保持し、次回の高速表示に備えます。

### 3.5. ローカライゼーション (L10n)
OSの言語設定に合わせた多言語対応（日英対応）を実装しています。
- **自動切り替え**: .NET標準のリソースファイル (`.resx`) を用い、Windowsの表示言語設定（`CurrentUICulture`）に連動してUIのテキストやメニューが自動で切り替わります（ユーザー設定不要）。
- **アプリ名のサニタイズ**: 「- 1つの実行中ウィンドウ」といったOS固有・言語依存のタスクバー表示テキストも、言語設定に合わせて正しく除去する仕組みを持っています。

## 4. クラス構造とデータフロー

1. アプリケーション起動時、`MainWindow` が非表示状態で生成され、`KeyboardHook` の初期化とシステムトレイへの登録が行われます。
2. ユーザーがWinキーを押し下げる。
3. `KeyboardHook` が **`WinKeyDown`** を即座に通知し、`MainWindow` は **`TaskbarReader.GetTaskbarApps()` のプリフェッチ**を開始する。
4. ユーザーがWinキーをそのまま長押し（600ms）し続ける。
5. `KeyboardHook` が長押しを検知し、`MainWindow` へ通知する。
6. `MainWindow` はまず **前回のキャッシュ結果を即座に表示**。
7. プリフェッチが完了次第、最新の `TaskbarApp` リストで表示を差し替え、アイコンを非同期で流し込む。
8. ユーザーがWinキーを離す。
9. `KeyboardHook` がリリースイベントを通知し、`MainWindow` がフェードアウトアニメーションを再生して非表示状態に戻る。

## 5. データの永続化

アプリケーションの各種設定は、Windowsレジストリを利用して保存されています。
- **スタートアップ設定**: `HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Run` に自身のパスを書き込むことで、Windows起動時の自動実行を実現しています。

## 6. 特記事項・今後の拡張性

- **UI Automationへの依存**: タスクバーの構造はWindowsのバージョンによって異なる場合があります。現状は TreeWalker と複数の戦略を組み合わせることで耐性を高めていますが、OSの大規模アップデート時には構造の再解析が必要になる可能性があります。
- **今後の展望**: 常駐時のメモリ使用量の更なる低減、およびアイコンのディスクへの永続キャッシュによる初回起動時の完全高速化などが検討可能です。
